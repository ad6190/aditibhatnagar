Glossary — Know the Language

Before diving in, align on what every term means. Bookmark this section — you'll reference it throughout.
Reference · 5 min overview
Core Terminology
Agent
An autonomous AI system that can plan, act, observe, and iterate to complete a task. Unlike a chatbot (one-shot Q&A) or a copilot (inline suggestions), an agent has a loop: it reads files, writes code, runs commands, inspects results, and keeps going until the job is done. Think: a junior developer that never sleeps.
Harness
The runtime environment that orchestrates an agent. It provides the agent loop, tool access, file I/O, terminal control, and safety guardrails. Examples: Cursor, Claude Code, Windsurf, GitHub Copilot, Aider, Cline.
Prompt
The instruction you give the agent. Can be a system prompt (persistent context the agent always sees), a user prompt (your request), or a custom prompt/command (a reusable template stored in your project). Quality of prompts directly determines quality of output.
Skill
A reusable, self-contained instruction set that teaches an agent how to do a specific task. Skills are markdown files (e.g., SKILL.md) stored in your project or shared repo. Example: a skill that knows how to create a PR following your team's conventions.
Hook
A shell command, HTTP endpoint, or LLM prompt that runs automatically at a specific point in the agent's lifecycle — e.g., before a file edit (auto-format), after a command runs (lint check), or on session start. Think: Git hooks, but for your AI agent.
Rule
A persistent instruction file that the agent always follows for a project or file type. Rules live in .cursor/rules/ or .cursorrules (Cursor) and CLAUDE.md (Claude Code). Think: team coding standards, but enforceable by AI.
Context Window
The total amount of text (measured in tokens) the model can "see" at once — your prompt, files, conversation history, tool outputs, everything. Current windows range from 128K to 1M+ tokens. When you exceed it, the model forgets earlier context.
MCP
Model Context Protocol — an open standard (by Anthropic) that lets agents securely connect to external tools, APIs, and databases through a uniform interface. Think of it as "USB for AI" — plug in a database, a Jira instance, or a Slack workspace, and the agent can use it.
Token
The fundamental unit of text for LLMs. Roughly 1 token ≈ 4 characters or ~¾ of a word in English. A 200K-token context window holds roughly 150,000 words — about 500 pages of a book, or a large codebase.
Open Weights
A model whose trained weight files are publicly available for download, fine-tuning, and self-hosting — but not necessarily the training data or full pipeline. This is what most people call "open source AI" (Llama, DeepSeek, Qwen), though strictly it's "open weights."
MoE
Mixture of Experts — an architecture where the model has many "expert" sub-networks but only activates a small fraction per query. This means a 400B-parameter model might only use 17B parameters per inference, making it fast and cheap. Nearly every frontier model in 2026 uses MoE.
Subagent
A child agent spawned by a parent agent to handle a subtask in isolation — with its own context window, often in a separate git worktree. Enables parallel work and prevents context pollution.
Worktree
A Git feature that lets you have multiple working directories from the same repo simultaneously. Agents use worktrees to work on tasks in isolation without interfering with each other or your main branch.
Grounding
The practice of injecting reality into the context window — retrieving your actual codebase patterns, current docs, and constraints before the agent generates code. Without grounding, agents hallucinate plausible-sounding solutions based on training data. With grounding, they build on what actually exists.
Background Agent
An AI agent that runs autonomously on cloud infrastructure, triggered by events (PR opened, issue created, schedule) rather than manual invocation. Unlike coding agents that run on localhost, background agents are decoupled from your device and session. You trigger it, walk away, and review the output later.
Compaction
The process of summarizing conversation history to free up context window space. The agent condenses what happened so far into a shorter summary, losing detail but preserving key decisions. Essential for long sessions. Claude Code: /compact. Cursor: automatic or new chat.
Checkpoint
A restore point in the agent's work — a saved state of files you can roll back to if the agent goes off track. Claude Code: /rewind. Cursor: built-in checkpoint system. Enables aggressive experimentation without risk.
Module 1

The Operator Mindset — Power Tools, Not Teammates

The single most important mental model shift: stop treating agents like junior developers. Start operating them like precision instruments.
~10 min in session
The Wrong Mental Model

Most developers treat AI agents like teammates: they wait for the agent to "understand," fix its code line-by-line, and fight context limits. Research shows developers without proper methodology are actually 19% slower with AI tools. The difference between 19% slower and 10x faster is operator skill.

THE CORE TRUTH AI agents aren't teammates — they're power tools. You don't collaborate with a table saw. You set it up precisely, feed it the right material, and verify the output. Same with agents. You are the operator. The quality of your setup determines the quality of the output.
Everything is Just Text

Here's the fundamental insight that demystifies AI agents: everything is just text flowing through a context window. No magic, no separate reasoning engine, no hidden state. When you see the agent "thinking" ("I should check the validation logic..."), that's not internal thought — it's text being generated in the context. The LLM is completely stateless. Its only "world" is the current context window.

LLM = Brain

A token prediction engine. Zero consciousness, zero intent. Samples from probability distributions learned from training data. Processes ~200K tokens of context. Predicts the next most probable token. That's all it does.

Agent Software = Body

The LLM alone can only generate text. The harness wraps it with tools: file I/O, terminal access, web search, MCP connections. The LLM predicts "I should read auth.ts" → the harness executes Read(src/auth.ts) → the result flows back as text.

The Four-Phase Operator Workflow

This is the systematic approach used in production environments. Every agent interaction follows this cycle:

1. Research → 2. Plan → 3. Execute → 4. Validate
Phase What You Do What the Agent Does Key Principle
1. Research Ground the agent in your codebase and domain before it acts Reads files, searches code, explores architecture Grounding — inject reality into context. Without it, agents hallucinate.
2. Plan Review the agent's proposed approach before execution Proposes architecture, identifies files to change, outlines steps Exploration vs. Exactness — explore when uncertain, be directive when clear.
3. Execute Let the agent implement, with checkpoint rhythm Writes code, runs tests, iterates on failures Supervised or autonomous — based on trust and task criticality.
4. Validate Verify against your mental model, then iterate or regenerate Reviews its own work in fresh context, runs final checks Iterate vs. Regenerate — patch small issues, regenerate if fundamentally wrong.
Grounding — The Bridge Between Training Data and Reality

Grounding is how you inject reality into the context window. You retrieve relevant information — your codebase patterns, current docs, constraints — and feed it to the agent before generation. Without grounding, agents build solutions from training data patterns that may not match your system.

Code Grounding

Before generating, have the agent read your existing code: "How is authentication handled in our codebase?" This populates context with your actual patterns, not generic ones.

Evidence Requirement

Always require evidence: file paths, line numbers, actual values. The agent cannot provide evidence without reading your code — this forces grounded responses instead of hallucinated ones.

Prompting as Pattern Completion

Prompting is not conversation. It's pattern completion. You draw the beginning of the pattern you want the model to complete. Skip pleasantries — "please" and "thank you" dilute signal. Be specific, be structured, be explicit.

// BAD: vague, conversational "Hey, can you please help me add some authentication to my API? I think JWT might work well. Thanks!" // GOOD: structured, constrained, with evidence requirements "Add JWT authentication to the Express API. - Read src/middleware/ to understand existing middleware patterns - Use jsonwebtoken library (already in package.json) - Protect all /api/v1/* endpoints EXCEPT /api/v1/auth/login - Token expiry: 24h. Payload: userId + role. - Return 401 for missing/invalid tokens - Do NOT modify existing session middleware - Provide file paths and line numbers for all changes"
THE THREE OPERATOR ERRORS
  • Anthropomorphizing: Assuming agents "know" things, "care" about outcomes, or "understand" your intent. They don't. They predict tokens.
  • Trusting without verifying: Accepting generated code without running it. The agent is better at finding issues in code than generating perfect code on the first try.
  • Context neglect: Not investing in rules files, not grounding before generation, not compacting. The context window is the agent's entire world — if it's messy, output is messy.
Tests as Guardrails

Build a sub-30-second smoke test suite covering critical junctions: core user journey, auth boundaries, database connectivity. Run it after each agent task, not after 20 changes. Codify this in your CLAUDE.md or rules file so agents run smoke tests automatically.

YOU ARE THE CIRCUIT BREAKER AI agents are amplifiers — of your architectural clarity, your testing discipline, your code patterns. Good or bad, they compound what exists. Research shows AI code has 8x more duplication — not because agents create it, but because they amplify existing patterns. Every accepted line becomes pattern context for future agents. Your engineering judgment determines which direction the exponential curve bends.

Foundations — Primitives & the Agent Loop

What are agentic coding primitives, how the agent loop works, and why the harness matters more than the model.
~15 min in session
The Spectrum: Chatbot → Copilot → Agent
💬

Chatbot

One-shot Q&A. You ask, it answers. No tool access, no iteration, no memory across turns. ChatGPT in a browser tab.

✨

Copilot

Inline code suggestions as you type. Reactive, not proactive. Sees the current file but doesn't explore the codebase. GitHub Copilot's tab-completion mode.

🤖

Agent

Autonomous loop: Plan → Act → Observe → Reflect → Iterate. Has tools (file read/write, terminal, web, MCP). Works across files, runs tests, self-corrects. Cursor Agent, Claude Code, Copilot Agent.

The Agent Loop
User Prompt → Plan → Use Tools → Observe Results → Reflect → Complete / Iterate
The agent keeps looping until the task is done (or it hits guardrails)

The key insight: the model is just one component. The harness provides the loop, the tools, the safety rails, and the context management. A mediocre model in a great harness often outperforms a great model in a bad harness.

What is a Harness?

A harness is the runtime that wraps a model and gives it agency. It controls:

  • The Agent Loop — how the model plans, acts, and iterates
  • Tool Access — which tools the agent can use (file I/O, terminal, web, MCPs)
  • Context Management — what the model sees in its window, when to compact, what to inject
  • Safety & Permissions — what the agent is allowed to do without asking (read? write? run commands?)
  • UI/UX — how you interact (IDE, CLI, web), how diffs are shown, how you approve changes

The Four Primitives

1. Rules / Instructions

  • What: Persistent guidelines the agent always follows.
  • Where: .cursor/rules/, CLAUDE.md, .github/copilot-instructions.md
  • Why: Encode your team's coding standards, architecture decisions, and project conventions so every agent follows them automatically.

2. Skills / Commands

  • What: Reusable instruction templates triggered with /command.
  • Where: .cursor/skills/, .claude/skills/, .claude/commands/
  • Why: Turn repeating workflows (PR creation, code review, test generation) into one-command actions.

3. Hooks

  • What: Automated scripts that fire at specific lifecycle events.
  • Where: .cursor/hooks.json, .claude/settings.json
  • Why: Auto-format on edit, lint after changes, block dangerous commands, enforce team policies — without human intervention.

4. MCP Connections

  • What: External tool integrations via the Model Context Protocol.
  • Where: .cursor/mcp.json, Claude Code MCP config
  • Why: Let agents read from your database, create Jira tickets, query Datadog, post to Slack — all through a standardized interface.
KEY INSIGHT The combination of Rules + Skills + Hooks + MCPs is what transforms a generic AI model into your team's AI developer. These primitives are version-controlled, shareable, and composable — they are the building blocks of an AI-augmented engineering team.

The Model Landscape — Closed, Open Weights & How to Choose

Understanding the models powering your agents — what "open weights" really means, the MoE revolution, and practical model selection.
~5 min in session · Deep-dive reference
Closed vs. Open Weights — What's the Difference?

Closed / Proprietary

  • Access: API only — you send requests, get responses
  • Control: Can't inspect weights, can't self-host, can't fine-tune
  • Examples: Claude (Anthropic), GPT-5 (OpenAI), Gemini (Google)
  • Strengths: Highest absolute quality, managed infrastructure, rich tool ecosystems
  • Trade-offs: Data leaves your network, vendor lock-in, cost at scale

Open Weights

  • Access: Download the model weights, run anywhere
  • Control: Self-host, fine-tune, quantize, run air-gapped
  • Examples: Llama 4 (Meta), DeepSeek V4 (DeepSeek), Qwen 3.5 (Alibaba), Mistral, Gemma 4
  • Strengths: Privacy, cost at scale (~50% cheaper), no vendor lock-in
  • Trade-offs: Ops burden, 1–3 points behind frontier on hardest benchmarks
IMPORTANT DISTINCTION "Open weights" ≠ "Open source." Open weights means the trained model files are public — you can run and fine-tune them. True open source would also include training data, preprocessing scripts, and full retraining capability. Most models called "open source" (Llama, DeepSeek, Qwen) are technically open weights.

Context Windows — Size Matters (But Strategy Matters More)

What is a Context Window?

Everything the model can "see" at once: your prompt, system instructions, file contents, conversation history, tool outputs. Measured in tokens (1 token ≈ 4 chars). When you exceed it, earlier content is silently dropped.

Current Window Sizes

  • 128K (~100K words) — Gemma, older models
  • 200K–256K — Most working models
  • 1M (~750K words) — Claude, DeepSeek, GPT-5
  • 10M — Llama 4 Scout (entire codebases!)
BIGGER ISN'T ALWAYS BETTER Models degrade on tasks requiring precise recall from the middle of very long contexts ("lost in the middle" problem). A 200K window with well-curated context often outperforms a 1M window stuffed with everything. This is why context engineering (Module 5) is a critical skill.
Practical Model Selection
  • Highest-stakes, user-facing: Closed frontier models (Claude Opus, GPT-5.5)
  • High-volume internal work: Open-weight models via API (DeepSeek V4-Flash at $0.28/M tokens)
  • Privacy-sensitive or air-gapped: Self-hosted open weights (Qwen, Llama, Mistral)
  • The practical reality: Most serious teams run a hybrid stack — closed for precision, open for volume

Module 5

Context Window Engineering

How LLMs actually "see" your codebase, strategies for managing context, and avoiding the most common failure modes.
~10 min in session
How the Agent Sees Your Code

When you ask an agent to work on a task, it doesn't "know" your codebase the way you do. It builds understanding through its context window — a fixed-size buffer that holds everything: your prompt, system instructions, file contents it reads, conversation history, and tool outputs.

Context Window (e.g., 200K tokens)
├── System Prompt + Rules ~2–5K tokens
├── Conversation History ~5–50K tokens
├── Files Read by Agent ~10–100K tokens
├── Tool Outputs ~5–30K tokens
├── MCP Responses ~1–10K tokens
└── Remaining for Reasoning ← this shrinks!
The Five Context Failure Modes

1. Context Overflow

Window is full. Earlier messages get silently dropped. Agent "forgets" requirements you stated 20 messages ago. Fix: Use /compact regularly. Start new sessions for new tasks.

2. Context Pollution

Irrelevant files, failed attempts, and dead-end conversations clutter the window. Agent gets confused. Fix: Be selective with @file references. Use Plan mode to narrow scope first.

3. Lost in the Middle

Information in the middle of a long context gets less attention than the beginning and end. Fix: Put critical instructions at the start (rules) and end (your prompt) of context.

4. Stale Context

Agent references files it read earlier, but they've changed since. Fix: Have the agent re-read files before making changes. Use hooks to re-validate.

5. Missing Context

Agent doesn't know about relevant files/conventions it hasn't been shown. Fix: Use rules files, @-mentions, codebase search, and project-level rules/CLAUDE.md.

Context Management Strategies
Strategy How When
Compact regularly Claude Code: /compact. Cursor: auto-compacts, or start new chat. Long sessions; after completing a subtask; when agent starts repeating itself
One task, one session Start a fresh agent session for each distinct task Always. Don't ask the agent to "also do X" after it's done Y.
Rules as persistent context Put critical info in rules/CLAUDE.md — always loaded, doesn't compete with conversation Coding standards, architecture decisions, dependency versions
Subagents for isolation Spawn child agents with fresh context for subtasks Complex tasks with independent sub-problems
Selective file references Use @file to add only the specific files needed, not entire directories Always. Be precise about what context the agent needs.
Plan → Build Use Plan mode first to identify which files and approaches matter Complex or ambiguous tasks
THE GOLDEN RULE OF CONTEXT Context engineering is the single highest-leverage skill for working with AI agents. The model is the engine; context is the fuel. Bad fuel = bad output, regardless of how powerful the engine is. Invest in your rules files, be intentional about what goes into the window, and compact aggressively.

MCP — Model Context Protocol

How agents securely connect to your internal tools, APIs, and databases. The universal adapter for AI tooling.
~10 min in session
What is MCP?

Model Context Protocol (MCP) is an open standard (created by Anthropic, now widely adopted) that defines how AI agents connect to external tools and data sources. Think of it as "USB for AI" — a universal plug that works across different harnesses and models.

Your Agent → MCP Client → MCP Server → Your Tools

PostgreSQL Jira / Linear Slack Datadog Internal APIs Git
MCP Architecture

MCP Server

Exposes Tools (functions the agent can call) and Resources (data the agent can read). A server wraps any API, database, or service. Can be a local process, a Docker container, or a remote service.

MCP Client

Built into the harness (Cursor, Claude Code, etc.). Discovers available servers, presents their tools to the agent, handles the protocol. You typically don't code this — the harness provides it.

Practical MCP Examples
Use Case MCP Server What the Agent Can Do
Database queries PostgreSQL MCP, MySQL MCP Query your database, inspect schemas, generate migrations
Project management Jira MCP, Linear MCP Read tickets, update status, create subtasks
Monitoring Datadog MCP, Sentry MCP Check error rates, read logs, correlate incidents
Communication Slack MCP Read channel messages, post updates, search history
Browser testing Playwright MCP, Browser MCP Navigate pages, click elements, take screenshots, verify UI
Internal APIs Custom MCP (you build it) Anything your API exposes — deploy, feature flags, user management
Setting Up MCPs
// Cursor: .cursor/mcp.json { "mcpServers": { "postgres": { "command": "npx", "args": ["@modelcontextprotocol/server-postgres", "postgresql://user:pass@localhost/mydb"] }, "jira": { "command": "npx", "args": ["@modelcontextprotocol/server-jira"], "env": { "JIRA_TOKEN": "${JIRA_TOKEN}" } } } } // Claude Code: add via CLI claude mcp add postgres -- npx @modelcontextprotocol/server-postgres postgresql://...
Building Your Own MCP Server

When your team has internal tools that don't have public MCP servers, build your own. An MCP server is a simple process that speaks the MCP protocol. SDKs are available in TypeScript, Python, Java, and Go.

# Minimal MCP server in TypeScript import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js"; const server = new McpServer({ name: "my-internal-api" }); server.tool("deploy", { env: z.enum(["staging", "prod"]) }, async ({ env }) => { const result = await triggerDeploy(env); return { content: [{ type: "text", text: result }] }; } );
TEAM STRATEGY Start with public MCP servers (database, browser). Then build one internal MCP for your most-used internal API. Share the configuration across the team via your agent repo (Module 8). The ROI compounds: every team member's agent gets the same capabilities.

Superpowers — Methodology for Agents
192K+ GitHub Stars 14 Agents Supported MIT License

What: A complete software development methodology shipped as composable skill files. Instead of letting agents "vibe code," Superpowers enforces a disciplined workflow: brainstorm → plan → implement (TDD) → review → ship.

Key Skills:

  • brainstorming — Socratic design refinement before writing any code
  • writing-plans — Detailed implementation plans with file paths and verification steps
  • test-driven-development — Enforces RED-GREEN-REFACTOR cycle; deletes code written before tests
  • subagent-driven-development — Dispatches subagents per task with two-stage review
  • systematic-debugging — 4-phase root cause analysis process
  • using-git-worktrees — Parallel development in isolated branches

How it works: Each skill file opens with an "Iron Law" — a non-negotiable rule with a table of rationalizations the agent might try to skip it. The agent reads these and follows the methodology.

Supports: Claude Code, Cursor, Codex, Copilot, Gemini CLI, Windsurf, Cline, and 7 others.

WHY THIS MATTERS Superpowers proves that the biggest improvement in AI code quality comes not from better models, but from better methodology enforcement. "Don't write code until you have a plan. Don't commit until tests pass." Simple rules, massive impact.
GitHub Spec Kit — Spec-Driven Development
104K+ GitHub Stars 30+ Agents Supported MIT License

What: An open-source toolkit that makes specifications the primary artifact — not code. "Specifications don't serve code — code serves specifications."

Constitution → Specify → Plan → Tasks → Implement
/speckit.constitution
Create or update project governing principles and development guidelines
/speckit.specify
Define what you want to build — requirements and user stories
/speckit.plan
Create technical implementation plans with your chosen tech stack
/speckit.tasks
Generate actionable task lists for implementation
/speckit.implement
Build the feature according to the plan

Maturity levels: Spec-first (write spec before prompting) → Spec-anchored (spec and code stay in sync, Spec Kit lives here) → Spec-as-source (spec is the only source, code is fully derived).

Beads — Persistent Agent Memory
20K+ GitHub Stars Dolt-Powered MIT License

What: A distributed graph issue tracker that gives agents persistent, structured memory. Replaces messy markdown TODO lists with a dependency-aware graph powered by a version-controlled SQL database (Dolt).

The Problem it Solves: Agents lose context between sessions. They can't track what they've done, what's blocked, or what's next. Beads is external long-term memory for agents.

bd ready
List tasks with no open blockers — what the agent should work on next
bd create
Create a new task with priority, dependencies, and metadata
bd prime
Sync the agent with current project state — inject context and persistent memories
bd remember
Store a persistent insight that survives session restarts and context compaction
bd update --claim
Atomically claim a task (prevents multi-agent collision)

Key features: Semantic compaction (summarizes old tasks to save context), hash-based IDs (no merge conflicts), multi-agent coordination via task claiming, and persistent memories via bd remember.

BMAD — Full-Lifecycle AI Framework
46K+ GitHub Stars 12+ Agent Personas 34+ Workflows Open Source

What: "Breakthrough Method for Agile AI-Driven Development" — a comprehensive framework that puts AI agents at every phase of the SDLC, not just coding. 12+ specialized agent personas (Business Analyst, Architect, Developer, UX Designer, etc.) with 34+ guided workflows.

Phase 1: Analysis → Phase 2: Planning (PRD) → Phase 3: Architecture → Phase 4: Implementation

Key idea: By Phase 4 (implementation), the agent has a complete specification chain — it doesn't guess, it executes against a well-defined plan that was shaped by requirements, architecture, and design decisions from earlier phases.

  • Superpowers — Execution: How to build it with discipline (TDD, reviews, etc.)

Why a Central Agent Repo?

Consistency

Every developer's agent follows the same coding standards, architecture patterns, and review processes. No more "works differently on my machine."

Compounding Returns

When one developer creates a great skill (e.g., "generate Spring Boot controller with our patterns"), the whole team benefits. Skills improve via PRs just like code.

Onboarding Accelerator

New team members get the team's AI capabilities on Day 1. Their agent already knows your conventions, tools, and workflows.

Knowledge Preservation

Team expertise gets encoded in skills and rules. When someone leaves, their knowledge doesn't.

Module 9

Debugging with AI-Native Tools

Translating your VS Code debugging skills to Cursor and Claude Code — plus new debugging paradigms that AI enables.
~5 min in session · Reference
The Debugging Paradigm Shift

Traditional debugging is you driving: you set breakpoints, you step through code, you inspect variables, you form hypotheses. AI-native debugging is evidence-based: you describe the problem, the agent forms hypotheses, instruments the code, runs experiments, and proves findings with reproducible evidence. Your job shifts from "driving" to "directing."

THE EVIDENCE RULE Never accept a fix without reproducible proof it works. The core debugging workflow is: REPRODUCE → GROUND → DIAGNOSE → FIX → VERIFY. The agent must re-run the reproduction after every fix attempt. If it can't prove the fix works, it iterates with a new hypothesis.
Cursor Debugging Workflow
  1. Switch to Debug mode — Cmd . → Debug. This activates systematic troubleshooting with hypothesis testing.
  2. Describe the bug — "When I submit the form with an empty email field, the app crashes with a NullPointerException instead of showing a validation error."
  3. Add context — Use @file to reference the relevant controller, service, and test files. Paste the error stack trace.
  4. Let the agent investigate — It will read files, trace the execution path, identify the root cause, and propose a fix.
  5. Review the fix — Cursor shows diffs inline. Accept or reject each change. The agent runs tests to verify.
  6. If the fix doesn't work — Tell the agent. It will form a new hypothesis and try again. Use Cmd Shift ⌫ to cancel if it's going in circles.
Claude Code Debugging Workflow
  1. Describe the bug with context — paste the error, reference the file, describe expected vs actual behavior.
  2. The agent traces the problem — reads relevant code, adds targeted logging, runs the failing test.
  3. Root cause identified — the agent explains what's wrong and why.
  4. Fix applied and verified — patches the code, runs tests. If tests fail, it iterates automatically.
  5. Cleanup — run /simplify to review the changes for quality and remove debugging artifacts.
  6. If it goes wrong — /rewind to roll back to a previous state and try a different approach.

PRO TIP: DEBUGGING AGENTS WITH AGENTS If an agent is producing bad output, don't debug the code — debug the context. Ask the agent: "What files have you read? What assumptions are you making? Show me your understanding of the architecture." Often the problem is missing or incorrect context, not a bad model.
Module 10

Background Agents & The Self-Driving Codebase

The maturity model from coding agents on localhost to autonomous agents in the cloud. Why individual speed-ups don't equal team speed-ups — and what does.
~5 min in session · Vision
The False Summit

Organizations that adopted coding agents saw individual engineers get faster. But cycle time didn't improve and DORA metrics stayed flat. Why? Gains compound with the individual, not the organization. Making one developer 2x faster doesn't halve your delivery time — it just means that developer is less busy while the bottleneck remains elsewhere.

THE LOCALHOST CEILING Coding agents run on your laptop. They fight over machine state, secrets become exposed in shell history, everything stops when the machine sleeps. Agents can't be triggered by events, can't run overnight, can't work across repos. Professional engineering requires decoupling agents from developer workstations.
Coding Agent vs. Background Agent
Coding Agent (Today) Background Agent (Next)
Runs on Your laptop / localhost Cloud infrastructure (sandboxed)
Triggered by You, manually Events: PR opened, issue created, schedule, webhook
Scope Single task, single repo Cross-repo, full SDLC
Developer role "In the loop" — watching, steering "On the loop" — reviewing output, calibrating
Governance Prompt-based (fragile) Infrastructure-enforced (identity, audit trails, permissions)
Stops when Machine sleeps, terminal closes Task completes or hits human review gate
The Maturity Model
1. Establish Primitives → 2. Find Bottlenecks → 3. Scale the Factory

Step 1: Establish Primitives

Sandboxed execution (isolated environments with full toolchains), governance (identity, permissions, audit trails enforced at runtime, not by prompt), context & connectivity (agents reach internal APIs, database replicas, private registries), and event-driven triggers.

Step 2: Find Bottlenecks

Survey developers. Map where time goes. Identify repetitive tasks with bounded blast radius: code review triage, CI failure analysis, dependency updates, CVE remediation, test coverage expansion, lint/standards enforcement.

Step 3: Scale the Factory

Move from "in the loop" (watching) to "on the loop" (reviewing, calibrating). Fleet coordination: parallel agents across hundreds of repos. Agent swarms: many agents, one outcome. Engineers shift from writing every line to designing systems and reviewing agent output.

Background Agent Use Cases (High ROI)
  • Dependency updates across repos: Fleet of agents updates packages, runs tests, opens PRs across hundreds of repositories
  • CVE remediation: Agent detects vulnerability, patches it, runs security tests, opens PR — within hours, not weeks
  • CI failure analysis: When CI fails on a PR, an agent triages the failure, determines if it's flaky or real, and posts a summary
  • PR code review triage: Agent does a first pass, flagging logic errors, security issues, and regressions before human reviewer sees it
  • Test coverage expansion: Agent identifies uncovered code, writes tests, opens PRs — overnight, continuously
  • Standards enforcement: Agent ensures all new code follows team conventions, linting rules, and architectural patterns
Getting Started with Background Agents

IN THE LOOP → ON THE LOOP The shift from "I use an agent to write code" to "agents handle the predictable parts while I focus on architecture, review, and judgment" is the most important organizational transformation. Start with the coding agent primitives (this course). Then look at background agents for the tasks that don't need a human at the keyboard.
Module 11

Making Your SDLC AI-Native

End-to-end transformation — from spec to deploy. AI-native PRDs, spec-first development, multi-agent orchestration, and legacy modernization.
~10 min in session
The AI-Native SDLC Pipeline
Spec / PRD → Architecture → Task Breakdown → Implementation → Review → Test → Deploy

AI agents participate at every stage, not just coding. The key shift: specs become the primary artifact, code becomes the output.

Spec-First Development

Traditional: idea → code → docs (maybe). AI-native: idea → spec → architecture → tasks → code → verification. The spec is machine-readable, versioned, and the single source of truth.

Why Specs First?

Agents work dramatically better when they have clear, structured context. A vague prompt produces vague code. A detailed spec produces precise implementation. The spec is the lever that multiplies agent quality.

The Spec Chain

Each document feeds the next: PRD → Architecture → Epics → Stories → Implementation. By the time the agent writes code, there's nothing left to guess about. This is what BMAD and Spec Kit formalize.

The Conductor Pattern — Multi-Agent Orchestration

For complex features, a single agent working sequentially is slow and error-prone. The Conductor Pattern uses one orchestrating agent that dispatches subtasks to specialized subagents working in parallel:

Conductor Agent

↓      ↓      ↓      ↓
API Agent   UI Agent   Test Agent   Docs Agent
Each in its own worktree, own context window, working in parallel

In practice:

  • Cursor: Use /multitask to run async subagents in parallel. Use /worktree for isolated branches. Tiled layout (v3.1+) shows agents side-by-side.
  • Claude Code: Use /agents to create subagents. Spawn background tasks with the SDK for CI/CD-scale orchestration.
  • Superpowers: subagent-driven-development skill handles this automatically — dispatches per-task agents with two-stage review.
Legacy Code Modernization

AI agents are uniquely powerful for legacy work — they can read and understand old code without emotional attachment or institutional baggage.

  • Dependency mapping: Agent analyzes imports, call graphs, and module boundaries to produce a dependency map
  • Refactor target identification: Agent identifies tightly-coupled modules, dead code, and modernization candidates
  • Incremental migration: Agent produces small, testable refactoring PRs — not a big-bang rewrite
  • Test generation: Agent writes tests for legacy code before refactoring (characterization tests)
  • Documentation generation: Agent reads code and produces documentation for undocumented modules
REALITY CHECK: WHERE AGENTS CAN'T BE TRUSTED (YET)
  • Security-critical code paths — always human-review
  • Complex distributed systems design — agents lack production intuition
  • Performance-critical optimizations — agents optimize for correctness, not necessarily performance
  • Cross-team coordination — agents don't understand organizational context
  • Final deployment decisions — always human-gated
The 50%+ Activation Goal

The goal isn't to replace developers — it's to make every developer 2-3x more productive by having agents handle the repetitive, well-defined parts of the SDLC. Today, most teams have <10% of their engineers actively using agents beyond basic autocomplete. The target: 50%+ of engineers using agents for meaningful tasks (not just tab completion) within one quarter.

How to get there:

  1. Week 1: Set up the agent repo. Install one harness (Cursor or Claude Code). Create 3 team rules.
  2. Week 2: Write 2-3 team skills for the most common workflows. Share them.
  3. Week 3: Add one MCP integration (database or project management). Run a team demo.
  4. Week 4: Each developer creates one personal skill for their workflow. Retrospective on what's working.
  5. Ongoing: Monthly skill review. Evolve the agent repo like a living codebase.
Module 12

Exercises & Next Steps

Hands-on labs to reinforce learning. Complete these after the session — each builds on the concepts covered.
Self-paced · 3-4 hours total
Exercise 1

Set Up Your Harness & Create Team Rules

Get your chosen harness configured with team-specific rules.

  1. Install Cursor or Claude Code (or both)
  2. Create a project rules file (.cursor/rules/team-standards.md or CLAUDE.md)
  3. Add rules for: coding style, naming conventions, preferred libraries, testing requirements
  4. Test it: ask the agent to create a new React component or Java controller and verify it follows your rules
  5. Share the rules file with your team via Git
Estimated: 30 min
Exercise 2

Create Your First Custom Skill

Encode a repeating workflow as a reusable skill.

  1. Identify a task you do at least weekly (PR creation, code review, test writing, etc.)
  2. Write a SKILL.md file that teaches the agent how to do it your way
  3. Place it in .cursor/skills/your-skill/SKILL.md or .claude/skills/your-skill/SKILL.md
  4. Test it by triggering the skill in a real task
  5. Iterate: what did the agent get wrong? Update the skill.
Estimated: 45 min
Exercise 3

Connect an MCP Server

Give your agent access to an external data source.

  1. Choose a data source: PostgreSQL, filesystem, or a public API
  2. Configure the MCP server in .cursor/mcp.json or via claude mcp add
  3. Ask the agent to query data, describe the schema, or generate a report
  4. Bonus: build a simple custom MCP server that wraps one of your internal APIs
Estimated: 45 min
Exercise 4

Context Window Debugging Challenge

Practice diagnosing and fixing context problems.

  1. Start a long session where you ask the agent to work on multiple unrelated tasks
  2. Observe when the agent starts "forgetting" earlier instructions or making inconsistent decisions
  3. Practice using /compact to recover, starting fresh sessions, and using subagents for isolation
  4. Experiment: same task with minimal context vs. maximal context — what produces better results?
Estimated: 30 min
Exercise 5

Spec-Driven Development with Spec Kit or BMAD

Practice the full spec-first workflow on a small feature.

  1. Install Spec Kit (pip install speckit) or BMAD
  2. Run /speckit.constitution or bmad-help to set up your project
  3. Choose a small feature (e.g., "add a user preferences page")
  4. Walk through the full flow: Specify → Plan → Tasks → Implement
  5. Compare the output quality vs. just prompting the agent directly
Estimated: 60 min
Exercise 6

Bootstrap the Team Agent Repository

Create the shared repo that your whole team will use.

  1. Create a new Git repo following the structure in Module 8
  2. Move your rules and skills from Exercises 1-2 into it
  3. Write an install.sh that sets up symlinks for Cursor and Claude Code
  4. Add a CONTRIBUTING.md with guidelines for adding new skills
  5. Invite your team to contribute — set up a PR review process for skills
Estimated: 45 min
